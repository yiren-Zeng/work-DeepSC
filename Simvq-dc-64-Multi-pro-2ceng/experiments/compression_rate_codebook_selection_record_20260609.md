# 压缩率与码本大小选择记录

生成时间：2026-06-09  
相关基准方案：`quality_v2_B_larger_cb128-16_unet2_ds8x2_k128-16`  
训练尺寸：`256 x 256`  
原定测试尺寸：`768 x 512`  
信道编码：LDPC，码率 `R_c = 1/2`  
调制方式：BPSK，`log2(M) = 1`

本文档记录之前围绕目标压缩率 `0.083` 的码本推导过程，并新增目标压缩率约 `0.042` 时的计算过程和可选码本组合。

## 1. 压缩率统一口径

这里的压缩率不是单纯的信源 BPP，而是带上信道编码率和调制阶数后的传输开销。

定义：

```text
compression_ratio =
    source_bits / (R_c * log2(M) * 3 * H * W)
```

其中：

```text
source_bits = 所有离散量化 index 转成 bit 后的总比特数
R_c         = 信道编码码率
log2(M)    = 每个调制符号承载的 bit 数
3          = RGB 三通道
H * W      = 测试图像像素数
```

在当前设定中：

```text
R_c = 0.5
BPSK: log2(M) = 1
```

所以：

```text
compression_ratio =
    source_bits / (0.5 * 1 * 3 * H * W)
```

对于原定测试尺寸 `768 x 512`：

```text
H * W = 768 * 512 = 393216
denominator = 0.5 * 1 * 3 * 768 * 512 = 589824
```

后续所有 `0.083` 和 `0.042` 的码本选择，默认都按这个 `768 x 512` 测试条件计算。

## 2. 网络层级与 token 数

共同结构：

```text
U-Net 层数 = 2
下采样倍率 = [8, 2]
累计下采样倍率:
  Layer 1 = 8
  Layer 2 = 16
BASE_CHANNELS = 128
embedding_dim_list = [256, 512]
```

测试 `768 x 512` 时，patch-wise token 数为：

```text
Layer 1 patch token:
  768 / 8 = 96
  512 / 8 = 64
  N1_patch = 96 * 64 = 6144

Layer 2 patch token:
  768 / 16 = 48
  512 / 16 = 32
  N2_patch = 48 * 32 = 1536
```

如果第 1 层使用 channel-wise CVQ：

```text
Layer 1 channel token = 第 1 层通道数 = 256
```

如果第 2 层也使用 channel-wise CVQ：

```text
Layer 2 channel token = 第 2 层通道数 = 512
```

令：

```text
b1 = log2(K1)
b2 = log2(K2)
```

其中 `K1`、`K2` 是两层码本大小。

## 3. 目标压缩率 0.083 的推导

之前目标为：

```text
compression_ratio = 0.0833333333
```

对应的 source bits：

```text
target_source_bits =
    0.0833333333 * 589824
  = 49152 bits
```

也可以先换成信源侧 BPP：

```text
source_bpp = compression_ratio * R_c * log2(M) * 3
           = 0.0833333333 * 0.5 * 1 * 3
           = 0.125 bits/pixel

target_source_bits = 0.125 * 768 * 512
                   = 49152 bits
```

## 4. 0.083 下的 patch-wise 方案

patch-wise 两层时：

```text
source_bits = 6144 * b1 + 1536 * b2
```

因为：

```text
6144 = 1536 * 4
```

所以：

```text
source_bits = 1536 * (4*b1 + b2)
```

要满足 `49152 bits`：

```text
1536 * (4*b1 + b2) = 49152
4*b1 + b2 = 32
```

因此，0.083 下 patch-wise 的整数 bit 组合必须满足：

```text
4*b1 + b2 = 32
```

可选组合示例：

| b1 | b2 | K1 | K2 | source_bits | 压缩率 | 备注 |
|---:|---:|---:|---:|---:|---:|---|
| 7 | 4 | 128 | 16 | 49152 | 0.0833333 | 原始方案，已采用 |
| 6 | 8 | 64 | 256 | 49152 | 0.0833333 | 更偏向第二层 |
| 5 | 12 | 32 | 4096 | 49152 | 0.0833333 | 第二层码本很大 |
| 4 | 16 | 16 | 65536 | 49152 | 0.0833333 | 第二层极大，不推荐 |

原始方案选择的是：

```text
K1 = 128, K2 = 16
b1 = 7, b2 = 4
```

计算：

```text
source_bits = 6144 * 7 + 1536 * 4
            = 43008 + 6144
            = 49152

compression_ratio =
    49152 / 589824
  = 0.0833333
```

这个配置的优点是：

- 第一层是高分辨率层，使用 `7 bit/index`，保留较多细节能力。
- 第二层是低分辨率层，使用 `4 bit/index`，保持整体结构信息。
- 两层码本大小适中，训练稳定。
- 训练 `256x256` 和测试 `768x512` 时压缩率一致，因为 patch-wise token 数随图像面积同比例变化。

## 5. 0.083 下的混合 CVQ 方案

混合 CVQ 指的是：

```text
Layer 1: channel-wise CVQ
Layer 2: patch-wise SimVQ
```

测试 `768 x 512` 时：

```text
Layer 1 token 数 = 256
Layer 2 token 数 = 1536
```

因此：

```text
source_bits = 256 * b1 + 1536 * b2
            = 256 * (b1 + 6*b2)
```

目标 source bits 是 `49152`，所以：

```text
256 * (b1 + 6*b2) = 49152
b1 + 6*b2 = 192
```

这是之前遇到的关键问题。

如果要求 `b1`、`b2` 都是整数，且 `K1=2^b1`、`K2=2^b2`，则必须满足：

```text
b1 + 6*b2 = 192
```

可选组合示例：

| b1 | b2 | K1 | K2 | 是否现实 |
|---:|---:|---:|---:|---|
| 108 | 14 | `2^108` | 16384 | 完全不可训练 |
| 96 | 16 | `2^96` | 65536 | 完全不可训练 |
| 78 | 19 | `2^78` | 524288 | 完全不可训练 |
| 60 | 22 | `2^60` | 4194304 | 完全不可训练 |
| 36 | 26 | `2^36` | 67108864 | 完全不可训练 |
| 24 | 28 | 16777216 | 268435456 | 仍然过大 |
| 18 | 29 | 262144 | 536870912 | 第二层过大 |
| 12 | 30 | 4096 | 1073741824 | 第二层过大 |

结论：

```text
在测试 768x512、Layer1 channel-wise + Layer2 patch-wise 的混合 CVQ 结构下，
如果仍要求压缩率为 0.0833333，则 b1 + 6*b2 = 192。
这会导致至少一个码本极端巨大，工程上不可训练。
```

这就是之前判断 `0.083` 下混合 CVQ 方案不可训练的核心原因。

## 6. 0.083 下的纯 CVQ / vanilla CVQ 方案

如果两层都做 channel-wise CVQ：

```text
Layer 1 token 数 = 256
Layer 2 token 数 = 512
```

则：

```text
source_bits = 256*b1 + 512*b2
            = 256 * (b1 + 2*b2)
```

目标为 `49152 bits`：

```text
256 * (b1 + 2*b2) = 49152
b1 + 2*b2 = 192
```

这同样不可行。示例：

| b1 | b2 | K1 | K2 | 是否现实 |
|---:|---:|---:|---:|---|
| 16 | 88 | 65536 | `2^88` | 完全不可训练 |
| 32 | 80 | `2^32` | `2^80` | 完全不可训练 |
| 64 | 64 | `2^64` | `2^64` | 完全不可训练 |
| 96 | 48 | `2^96` | `2^48` | 完全不可训练 |

结论：

```text
纯 channel-wise CVQ 在 0.083 目标下更不可行。
因为 channel-wise token 数太少，要达到 49152 source bits，只能把每个 token 的 bit 数推得极高。
```

## 7. 目标压缩率 0.042 的新增推导

如果目标压缩率改成：

```text
compression_ratio ≈ 0.042
```

对应的理想 source bits 为：

```text
target_source_bits =
    0.042 * 589824
  = 24772.608 bits
```

由于 `b1`、`b2` 必须是整数，实际 source bits 通常无法精确等于 `24772.608`，因此要选择最接近的整数 bit 组合。

一个常用近似目标是：

```text
compression_ratio = 0.0421006944
source_bits = 24832
```

因为：

```text
24832 / 589824 = 0.0421006944
```

## 8. 0.042 下的 patch-wise 方案可选码本

patch-wise 两层公式仍然是：

```text
source_bits = 1536 * (4*b1 + b2)
compression_ratio = source_bits / 589824
```

因为：

```text
589824 / 1536 = 384
```

所以：

```text
compression_ratio = (4*b1 + b2) / 384
```

目标 `0.042` 对应：

```text
4*b1 + b2 = 0.042 * 384 = 16.128
```

因此最接近的整数目标有两个：

```text
4*b1 + b2 = 16  -> compression_ratio = 16 / 384 = 0.0416667
4*b1 + b2 = 17  -> compression_ratio = 17 / 384 = 0.0442708
```

### 8.1 不超过 0.042 的 patch-wise 组合

满足：

```text
4*b1 + b2 = 16
```

可选：

| b1 | b2 | K1 | K2 | 压缩率 | 备注 |
|---:|---:|---:|---:|---:|---|
| 1 | 12 | 2 | 4096 | 0.0416667 | 第一层太弱 |
| 2 | 8 | 4 | 256 | 0.0416667 | 第一层偏弱 |
| 3 | 4 | 8 | 16 | 0.0416667 | 平衡但第一层仍较小 |

之前曾生成的 A 方案是：

```text
K1 = 8, K2 = 16
compression_ratio = 0.0416667
```

该方案严格不超过 `0.042`，但第一层码本小于第二层。

### 8.2 接近但略高于 0.042 的 patch-wise 组合

满足：

```text
4*b1 + b2 = 17
```

可选：

| b1 | b2 | K1 | K2 | 压缩率 | 备注 |
|---:|---:|---:|---:|---:|---|
| 1 | 13 | 2 | 8192 | 0.0442708 | 第一层太弱 |
| 2 | 9 | 4 | 512 | 0.0442708 | 第一层偏弱 |
| 3 | 5 | 8 | 32 | 0.0442708 | 较均衡 |
| 4 | 1 | 16 | 2 | 0.0442708 | 第一层大于第二层，用户最终采用 |

用户最终指定的 A / patch-wise 方案是：

```text
K1 = 16
K2 = 2
b1 = 4
b2 = 1
compression_ratio = 0.0442708
```

计算：

```text
source_bits = 6144 * 4 + 1536 * 1
            = 24576 + 1536
            = 26112

compression_ratio =
    26112 / 589824
  = 0.0442708
```

选择原因：

```text
第一层是高分辨率层，对图像重建更重要；
因此选择 K1 > K2，即第一层码本更大、第二层码本更小。
```

## 9. 0.042 下的混合 CVQ 方案可选码本

混合 CVQ 公式：

```text
source_bits = 256*b1 + 1536*b2
            = 256 * (b1 + 6*b2)
```

因为：

```text
589824 / 256 = 2304
```

所以：

```text
compression_ratio = (b1 + 6*b2) / 2304
```

目标 `0.042` 对应：

```text
b1 + 6*b2 = 0.042 * 2304 = 96.768
```

因此最接近的整数目标为：

```text
b1 + 6*b2 = 97
compression_ratio = 97 / 2304 = 0.0421006944
```

另一个略低的整数目标为：

```text
b1 + 6*b2 = 96
compression_ratio = 96 / 2304 = 0.0416667
```

### 9.1 接近 0.04210 的混合 CVQ 组合

满足：

```text
b1 + 6*b2 = 97
```

可选：

| b1 | b2 | K1 | K2 | 压缩率 | 备注 |
|---:|---:|---:|---:|---:|---|
| 13 | 14 | 8192 | 16384 | 0.0421007 | 之前建议/采用过，第二层略大 |
| 19 | 13 | 524288 | 8192 | 0.0421007 | 第一层更大，但 K1 很大 |
| 25 | 12 | 33554432 | 4096 | 0.0421007 | K1 过大 |
| 31 | 11 | 2147483648 | 2048 | 0.0421007 | 不可训练 |
| 7 | 15 | 128 | 32768 | 0.0421007 | 第二层过大 |

之前一版 B/C 选择的是：

```text
K1 = 8192
K2 = 16384
b1 = 13
b2 = 14
compression_ratio = 0.0421007
```

计算：

```text
source_bits = 256 * 13 + 1536 * 14
            = 3328 + 21504
            = 24832

compression_ratio =
    24832 / 589824
  = 0.0421007
```

该方案的问题：

```text
K1 < K2
但第 1 层是高分辨率层，按重建重要性看，用户认为更希望 K1 > K2。
```

### 9.2 第一层更大的混合 CVQ 组合

如果要求：

```text
K1 > K2
也就是 b1 > b2
```

在接近 `0.042` 的范围内，可选组合有：

| b1 | b2 | K1 | K2 | b1 + 6*b2 | 压缩率 | 备注 |
|---:|---:|---:|---:|---:|---:|---|
| 16 | 13 | 65536 | 8192 | 94 | 0.0407986 | 用户最终采用，略低于 0.042 |
| 17 | 13 | 131072 | 8192 | 95 | 0.0412326 | 更接近 0.042，但 K1 更大 |
| 18 | 13 | 262144 | 8192 | 96 | 0.0416667 | 不超过 0.042，K1 较大 |
| 19 | 13 | 524288 | 8192 | 97 | 0.0421007 | 最接近 0.04210，但 K1 很大 |
| 20 | 13 | 1048576 | 8192 | 98 | 0.0425347 | 略高于 0.042 |
| 24 | 12 | 16777216 | 4096 | 96 | 0.0416667 | K1 过大 |
| 25 | 12 | 33554432 | 4096 | 97 | 0.0421007 | K1 过大 |

用户最终指定的 B/C 混合 CVQ 方案是：

```text
K1 = 65536
K2 = 8192
b1 = 16
b2 = 13
```

计算：

```text
source_bits = 256 * 16 + 1536 * 13
            = 4096 + 19968
            = 24064

compression_ratio =
    24064 / 589824
  = 0.0407986
```

选择原因：

```text
1. 第一层码本大于第二层，符合“高分辨率层更重要”的直觉。
2. K1=65536 虽大，但仍比 524288、1048576 等组合更可训练。
3. 实际压缩率 0.04080，略低于 0.042，属于更严格的压缩。
```

## 10. 0.042 下的纯 CVQ 方案可选码本

如果两层都 channel-wise：

```text
source_bits = 256*b1 + 512*b2
            = 256 * (b1 + 2*b2)

compression_ratio = (b1 + 2*b2) / 2304
```

目标 `0.042` 对应：

```text
b1 + 2*b2 = 96.768
```

最接近：

```text
b1 + 2*b2 = 97
compression_ratio = 0.0421007
```

可选组合示例：

| b1 | b2 | K1 | K2 | 压缩率 | 是否建议 |
|---:|---:|---:|---:|---:|---|
| 65 | 16 | `2^65` | 65536 | 0.0421007 | 不可训练 |
| 57 | 20 | `2^57` | 1048576 | 0.0421007 | 不可训练 |
| 49 | 24 | `2^49` | 16777216 | 0.0421007 | 不可训练 |
| 33 | 32 | `2^33` | `2^32` | 0.0421007 | 不可训练 |
| 19 | 39 | 524288 | `2^39` | 0.0421007 | 第二层不可训练 |

结论：

```text
纯 CVQ 的 token 数太少；
无论 0.083 还是 0.042，都需要极大的 b1/b2 才能达到目标 source_bits。
因此当前工程中不建议做两层全 channel-wise CVQ。
```

## 11. 训练 256x256 与测试 768x512 的区别

对于 patch-wise：

```text
token 数随 H*W 同比例增长
```

因此同一组 `K1,K2` 在 `256x256` 和 `768x512` 下压缩率相同。

例如 A 最终方案：

```text
K1=16, K2=2
测试 768x512 压缩率 = 0.0442708
测试 256x256 压缩率 = 0.0442708
```

对于混合 CVQ：

```text
Layer 1 channel token 数固定为 256
Layer 2 patch token 数随图像面积变化
```

因此同一组 `K1,K2` 在不同测试分辨率下压缩率不同。

以 B/C 最终方案 `K1=65536,K2=8192` 为例：

### 11.1 在 768x512 测试时

```text
Layer 1 bits = 256 * 16 = 4096
Layer 2 bits = 1536 * 13 = 19968
source_bits = 24064

compression_ratio =
    24064 / (0.5 * 1 * 3 * 768 * 512)
  = 0.0407986
```

### 11.2 在 256x256 测试时

```text
Layer 1 bits = 256 * 16 = 4096
Layer 2 patch token = 16 * 16 = 256
Layer 2 bits = 256 * 13 = 3328
source_bits = 7424

compression_ratio =
    7424 / (0.5 * 1 * 3 * 256 * 256)
  = 7424 / 98304
  = 0.0755208
```

所以，B/C 最终方案在：

```text
768x512 测试时压缩率 = 0.0407986
256x256 测试时压缩率 = 0.0755208
```

这也是为什么在 `Kodak-256-transform-resize` 上测试时，B/C 性能明显提升：除了 CVQ codeword shape 与训练尺寸一致外，实际压缩率也从 `0.04080` 提高到了 `0.07552`。
